RAG 三代架构 演进

从简单拼接到组件化编排，RAG 架构在三年内经历了三次范式升级

G1

Naive RAG（2020）

Query → 向量检索 Top-K → 拼接 Context → LLM 生成
简单但召回质量低，无 Rerank，一次检索定生死

检索-拼接-生成
G2

Advanced RAG（2023）

+ Pre-Retrieval（Query 改写 / 检索路由）
+ Post-Retrieval（Rerank / 上下文压缩）
前后两端优化，召回率显著提升

前后优化
G3

Modular RAG（2024）

组件化拆分 · 支持任意组合编排
可跳过检索 / 多次检索 / 自适应检索
代表：Self-RAG（自判断是否需要检索）· CRAG（检索置信度评估）

组件化编排
进阶知识 · 检索路由 & 质量瓶颈

🔀检索路由（Retrieval Router）

根据 Query 类型选择检索策略
▾

不是所有问题都该走向量检索。检索路由根据 Query 特征自动选择最优路径：

Query 类型路由策略示例
事实型关键词检索 / 知识图谱"2024年GDP是多少"
语义型向量检索"怎么理解RAG的价值"
聚合型多路检索 + 合并"对比三家的RAG方案"
无需检索直接 LLM"帮我翻译这段话"
PM 要点：检索路由是 Advanced RAG 到 Modular RAG 的关键升级，面试高频问法："你们的 RAG 系统怎么决定用什么检索方式？"

⚠️RAG 质量瓶颈与 Lost in the Middle

知道天花板在哪，才能对症下药
▾
瓶颈现象解法方向
检索噪声召回了不相关的 ChunkRerank / 检索路由 / 更好的切分
Lost in the MiddleLLM 只关注 Context 开头和结尾，中间信息被忽略控制 Top-K 数量 / 重排把关键信息放两端
知识冲突检索到的多个 Chunk 说法矛盾时间戳标注 / 来源优先级 / 冲突检测
幻觉残留LLM 在有 Context 的情况下仍编造引用溯源 / Faithfulness 评分 / 兜底策略
面试高频："你们 RAG 遇到过什么质量问题？怎么解决的？" — 答 Lost in the Middle + Rerank 策略即可加分。

面试真题

高频 面试真题 汇总

数据项目数据来源有哪些？
▾
企业内部/外网公开/付费外采/多轮对话QA/网盘与API系统，根据业务场景定义知识边界与优先级
元数据怎么处理元数据的？
▾
将解析后文本与采集阶段记录的元数据（来源、采集时间）绑定，确保检索结果可溯源
排版复杂排版的阅读顺序怎么解决？
▾
使用腾讯DREAM模型（图像特征提取→元素聚合→并行生成）或DocTR文档布局分析，按"左栏→右栏"顺序提取
成本RAG中怎么有效降低成本？
▾
Chunk切分成三级结构，通过检索二级片段，将Top-K中同属一级片段的进行替换合并，减少Token消耗
PM0到1项目PM应侧重什么？
▾
构建核心功能，而非知识库层面技术细节。PM应全程参与，与业务/技术/数据/工程紧密合作
架构RAG 三代架构有什么区别？
▾
Naive RAG(2020)：检索-拼接-生成，简单但召回差。Advanced RAG(2023)：加 Pre/Post-Retrieval 优化(Query改写+Rerank)。Modular RAG(2024)：组件化编排，可跳过/多次/自适应检索，代表方案 Self-RAG 和 CRAG
质量RAG 回答质量不好怎么排查？
▾
四步排查：①检索噪声→优化切分+加Rerank ②Lost in the Middle→控制Top-K+重排关键信息到两端 ③知识冲突→时间戳标注+来源优先级 ④幻觉残留→引用溯源+Faithfulness评分+兜底策略
路由你们RAG怎么决定用什么检索方式？
▾
通过检索路由模块：事实型Query走关键词/知识图谱，语义型走向量检索，聚合型走多路合并，翻译/闲聊等无需检索直接给LLM。路由器可用规则或分类模型实现

术语速查

核心 术语表

点击术语卡片查看详细解释，掌握 RAG 领域关键概念

RAG
检索增强生成
Retrieval-Augmented Generation，给大模型接入外部知识库。流程：构建知识库 → 检索相关片段 → 增强 Prompt → LLM 生成。解决时效性、幻觉、安全性、专业知识四大限制。
Embedding
文本向量化
将文本转换为高维数学向量的过程，使计算机能够通过向量距离衡量语义相似度。推荐模型：BGE-M3（多语言多粒度）。是 RAG 离线阶段 Pipeline E 的核心步骤。
Token
模型处理最小单元
LLM 的最小处理单元。1个中文字 ≈ 1.5-2个 Token，1个英文单词 ≈ 1个 Token。Token 数量直接影响 API 调用费用和上下文窗口限制。
Chunk
文本切片
文本分割后的片段，推荐 300-800 字，重叠 10-20%。分割策略包括：固定长度、语义感知、文档结构、关键词、多粒度、混合策略。是 Pipeline D 的输出。
VectorDB
向量数据库
专门存储和检索向量数据的数据库（如 Milvus、Pinecone）。与传统 RDBMS/NoSQL 的精准匹配不同，VectorDB 支持语义模糊检索，能找到语义相似但文字不同的内容。
CoT
思维链推理
Chain of Thought，让 LLM 分步推理以提升准确率的技术。代表：DeepSeek-R1。属于模型演进 3.0 阶段的涌现能力，是 Prompt 工程的重要策略之一。
ICL
上下文学习
In-Context Learning，将示例放入 Prompt 上下文中，让模型参照模板作答。无需训练，通过 Few-shot 样本即可引导模型输出格式和风格。
Transformer
自注意力架构
Google 2017 年提出的神经网络架构，通过自注意力机制实现并行计算，解决了 RNN 的序列依赖问题。是所有现代 LLM 的基础架构。
SFT
监督微调
Supervised Fine-Tuning，使用人工标注数据对预训练模型进行微调，使其适应特定任务。结合 RLHF 对齐后诞生了 ChatGPT。作为 RAG 的替代方案之一，成本高但响应快。
Top-K
检索结果排序截取
在检索阶段，根据相似度 Score 对结果排序后取前 K 个最相关的 Chunk。K 值的选择影响召回率和精度的平衡，PM 质量标准：召回率≥90%，准确率≥85%。
Graph RAG
知识图谱增强
基于知识图谱（实体-关系-实体三元组）的 RAG 变体。适用于政策穿透、合规审查、科研追踪、故障分析等需要关系推理的场景。架构：Question → Encoder → Graph DB → LLM。
Temperature
生成温度参数
控制 LLM 生成随机性的参数。调大（如 0.8-1.0）→ 输出更具创意和多样性；调小（如 0.1-0.3）→ 输出更精确和确定性。RAG 场景通常使用较低 Temperature。
Naive RAG
第一代 RAG 架构
RAG 第一代架构（2020）。流程：Query → 向量检索 Top-K → 拼接 Context → LLM 生成。优点是实现简单成本低，缺点是无 Rerank、无 Query 优化、一次检索定生死，无法处理多跳问题。
Advanced RAG
第二代 RAG 架构
RAG 第二代架构（2023）。在 Naive RAG 基础上加入 Pre-Retrieval（Query 改写/检索路由）和 Post-Retrieval（Rerank/上下文压缩）两端优化，召回率显著提升。
Modular RAG
第三代 RAG 架构
RAG 第三代架构（2024）。将 RAG 拆为可独立替换的模块（检索器/路由器/生成器等），支持任意组合编排。可跳过检索、多次检索、自适应检索。代表方案：Self-RAG、CRAG。
Self-RAG
自适应检索增强
Self-Reflective RAG，模型自己判断"这个问题需不需要检索"。如果自身知识足够就直接回答，不够再触发检索。属于 Modular RAG 的代表方案，减少不必要的检索开销。
CRAG
检索置信度评估
Corrective RAG，在检索后评估结果置信度：高置信度直接用，低置信度触发网页搜索补充，模糊区域做知识精炼。属于 Modular RAG 的自纠错机制。
Lost in the Middle
中间信息遗忘
LLM 处理长 Context 时倾向于关注开头和结尾，中间的信息容易被忽略。解法：控制 Top-K 数量、Rerank 把关键 Chunk 排到两端、压缩冗余 Context。是 RAG 质量的典型瓶颈之一。
检索路由
Query 智能分发
根据 Query 类型自动选择检索策略的决策模块。事实型走关键词/知识图谱，语义型走向量检索，聚合型走多路合并，无需检索的直接给 LLM。是 Advanced → Modular RAG 的关键升级。
